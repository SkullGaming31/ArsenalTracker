Arsenal Tracker

A lightweight Vue + TypeScript app to track Warframe warframes & weapons — collection status, mastery progress, and quick dashboard stats.

Developer notes: this is a minimal landing page used by GitHub Pages. For the full app sources, run locally with npm install and npm run dev, or see the repository on GitHub.

Developer / Maintainer notes

  • Active working branch: enhancement/tests-lint-2025-10-27 — contains recent typing, unit tests and lint fixes.
  • Local persistence: the app stores overrides in localStorage under the key arsenaltracker.v1. There is a migration path handled in src/stores/collection.ts.
  • Debounced save: overrides are persisted with a ~200ms debounce to avoid UI freezes when toggling many items. In test environments persistence is immediate.
  • Tests & linting: use the project scripts to run checks locally:
    npm run lint
  npm run test:unit
  • ESLint notes: generated/minified files in docs/assets and Node utility scripts in scripts/ commonly produce lint warnings (commonjs require() usage and minified expression noise). It's recommended to add targeted ESLint overrides for docs/assets/** and set env: node for scripts/** to avoid noisy errors.
  • Next recommended improvements (high-impact):
    1. Type the importer (src/lib/importer.ts) and WarframeCard.vue to remove remaining @typescript-eslint/no-explicit-any issues.
    2. Apply small ESLint overrides for non-runtime/generated files so linting focuses on app code.
    3. Run the test suite and fix any remaining vitest lint rule failures that request exact-argument assertions (use toHaveBeenCalledWith(...) or expect.any(...) where appropriate).
  • Changelog: see CHANGELOG.md at the repository root for a summary of recent changes.

Chat / Bot usage

If you're interacting with this repository through a chat bot (for example to request code edits, tests, or explanations), the following guidance will help the bot provide useful and safe changes.

  • Scope your request: tell the bot the exact file(s) to change or the feature to implement (e.g., "Update src/lib/importer.ts to use a typed ParsedRow").
  • Include the branch context if relevant: e.g., enhancement/tests-lint-2025-10-27 — the bot can commit changes to that branch.
  • Testing: ask the bot to run linters and unit tests after edits. Example: "run ESLint and Vitest and report failures" — the bot will run commands and return results.
  • Example prompts: 
    - "Convert all remaining any types in src/lib/importer.ts to concrete types and run tsc --noEmit"
                - "Add an ESLint override for scripts/** to allow Node-style require() and re-run lint"
                - "Update docs/index.html to include bot usage guidance"
  • What the bot can do for you:
    • Make small code edits and run quick tests (unit/lint/typecheck).
    • Add or update tests and type definitions.
    • Propose configuration changes (ESLint, Vitest) and apply low-risk updates.
  • Tips for reliable responses:
    1. Provide a clear single objective per request (smaller commits are easier to verify).
    2. When asking for code changes, prefer to identify the precise file and a short description of the desired behavior.
    3. Ask the bot to run failing commands (lint/tests) and include the output when requesting follow-ups — this speeds debugging.
